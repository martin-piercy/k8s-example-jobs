# GB200 Kubernetes Example Jobs

Example Kubernetes manifests for running large-scale GPU workloads on NVIDIA GB200 NVL72 infrastructure. These are real job definitions that have been tested on multi-rack GB200 clusters with 4 GPUs per node.

All jobs use **JobSet** for multi-node coordination, **DRA ComputeDomains** for NVLink domain management, and **RDMA shared devices** for InfiniBand networking.

## Prerequisites

| Component | Purpose |
|---|---|
| [JobSet controller](https://github.com/kubernetes-sigs/jobset) | Multi-node indexed job orchestration |
| [NVIDIA GPU Operator](https://docs.nvidia.com/datacenter/cloud-native/gpu-operator/) | DRA driver, device plugin, GPU scheduling |
| [NVIDIA Network Operator](https://docs.nvidia.com/networking/display/cokan10) | RDMA/InfiniBand support, Multus, SRIOV device plugin |
| ComputeDomain controller | IMEX/NVLS multi-node NVLink coordination |
| [KAI Scheduler](https://github.com/kai-scheduler/KAI-Scheduler) | Topology-aware gang scheduling (only for `nemotron-kai-340b-tp8-jobset.yaml`) |

## Jobs

### NCCL Benchmarks

| File | Nodes | GPUs | Description |
|---|---|---|---|
| `GB200/NCCL_36n_rdmaShared.yaml` | 36 | 144 | `all_reduce_perf_mpi` across 2 racks, SSH-based MPI launcher |
| `GB200/NCCL_Single_rack_rdmashared.yaml` | 18 | 72 | `all_reduce_perf_mpi` on a single rack |

Both NCCL tests install `openssh-server` at runtime, build a hostfile from JobSet DNS, and use `mpirun` to launch the benchmark. Rank 0 is the launcher; all other pods run sshd and wait for work.

**Prerequisite:** Create the SSH key secret before deploying:
```bash
ssh-keygen -t ed25519 -f /tmp/nccl-ssh -N ""
kubectl create secret generic nccl-ssh-keys \
  --from-file=id_ed25519=/tmp/nccl-ssh \
  --from-file=id_ed25519.pub=/tmp/nccl-ssh.pub \
  --from-file=authorized_keys=/tmp/nccl-ssh.pub
```

## Before You Deploy — Cluster-Specific Values

These manifests contain placeholder values and cluster-specific defaults that you **must** adjust for your environment:

### 1. NVLink Clique ID (required)

Replace `<YOUR_CLIQUE_ID>`, `<RACK_A_CLIQUE_ID>`, and `<RACK_B_CLIQUE_ID>` with your actual NVLink domain identifiers:

```bash
kubectl get nodes -L nvidia.com/gpu.clique
```

Single-rack jobs need one clique ID. Multi-rack jobs (NCCL 36-node, Nemotron KAI) need two — one per rack.

### 2. NetworkAttachmentDefinitions

The Multus annotations reference NAD names like `ipoib-rdma-shared-device-default-{a,b,c,d}-su-1`. These are generated by the NVIDIA Network Operator based on your fabric configuration. Check yours:

```bash
kubectl get net-attach-def
```

### 3. InfiniBand HCA Devices

`NCCL_IB_HCA` is set to `mlx5_0,mlx5_1,mlx5_4,mlx5_5` (4-port ConnectX configuration). Verify your device names:

```bash
ibstat | grep "CA '"
```

### 4. RDMA Shared Device Resources

Resource requests use `rdma/rdma_shared_device_{a,b,c,d}`. These are defined by the SRIOV/RDMA device plugin in your Network Operator config. If your device plugin uses different resource names, adjust accordingly.

### 5. HuggingFace Token (GPT-OSS 120B only)

`gpt-oss-120b.yaml` references a `mpiercy-hf-token` secret. Create it with your own token if the model requires gated access:

```bash
kubectl create secret generic mpiercy-hf-token --from-literal=token=hf_xxxxx
```

## Usage

```bash
# Deploy a job
kubectl apply -f GB200/qwen-72b.yaml

# Watch pods come up
kubectl get pods -l jobset.sigs.k8s.io/jobset-name=mpiercy-qwen72b -w

# Follow the master (rank 0) logs
kubectl logs mpiercy-qwen72b-w-0-0 -f

# Clean up
kubectl delete -f GB200/qwen-72b.yaml
```

## GB200 NVL72 Topology Notes

These jobs are designed for the GB200 NVL72 topology: **4 GPUs per node** connected via NVLink within a rack (up to 72 GPUs per NVLink domain). Key architectural choices:

- **ComputeDomains** enable IMEX daemon coordination for multi-node NVLink (NVLS)
- **`nvidia.com/gpu.clique`** labels identify NVLink domains — pods in the same clique can use NVLink for inter-node communication
- **`topologySpreadConstraints`** ensure even pod distribution across the clique
- **4 RDMA devices per node** map to the 4 InfiniBand ports on each GB200 tray
- **`runtimeClassName: nvidia`** enables the DRA GPU driver (used instead of the legacy device plugin)

For clusters without DRA/ComputeDomain support, remove the `resourceClaims`, `claims`, and `ComputeDomain` resources and use the standard `nvidia.com/gpu` device plugin resource instead.
